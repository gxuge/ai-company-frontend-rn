# Figma 转代码开发工作流

此 Skill 定义了使用 Figma MCP 提取设计并将其转换为可用前端代码的标准流程。严格遵守这些规则可以确保代码的可维护性、正确的资源管理以及整洁的代码架构。

## 1. 资源管理（第一步，至关重要）
在编写任何组件代码之前，您**必须**处理设计中存在的所有视觉资源（图片、图标、SVG）。

- **确认页面名称：** 如果用户尚未提供正在构建的页面/组件的特定名称，您**必须首先向用户询问预期的页面名称**，然后才能继续。
- **创建资源目录：** 在 `assets/images/[页面名称]/` 下专门为该页面的资源创建一个文件夹。
- **下载资源：** 将 Figma MCP 输出中的所有图片和图标（通常使用 `http://localhost:3845/assets/...` 或类似的本地资源 URL）下载到这个新创建的目录中。
- **本地路径：** 您生成的代码中的所有图片和图标引用**必须**使用指向此本地 `assets/images/[页面名称]/` 目录的相对路径。绝对不要在最终代码中保留本地/临时的 Figma URL。
- **SVG 图标显示：** SVG 图标获取后，要确保在 HTML 中能够被正确访问到，在使用时需设定好合适的样式属性，确保图标整齐显示，不出错或变形。

## 2. 目录规范与结构分离
根据生成的类型决定存放位置，并严格分离结构和样式文件，保持代码库的清晰与可维护。

- **页面级规范：** 如果生成的是完整页面，必须将其存放在页面目录中（如 `pages/` 或 `app/pages/`），**且务必在路由系统或主导航中添加该页面的入口链接**。
- **组件级规范：** 如果生成的是独立组件，必须将其存放在 `components/` 目录下，并为其**新建一个同名文件夹**（名称由用户指定，例如 `components/[名称]/`），将有关代码及样式统一收纳其中。
- **组件主文件：** 创建核心的 `.tsx`（或其他适当扩展名）文件，其中**仅**处理 DOM 结构、属性映射及必要交互逻辑。
- **独立样式文件：** 根据项目配置单独创建一个样式文件（如 `.scss`，或 React Native 的 `StyleSheet` 对象等），专门用于存放该模块的所有视觉约束。
- **导入与映射绑定：** 确保无论页面还是组件，都能正确导入其专属样式，并将样式类或对象精确绑定至所有对应元素或标签上。

## 3. 字体排版：使用系统默认字体
Figma 设计图通常会指定非常具体的自定义字体。为了提高性能和兼容性，您必须无视这些自定义字体，转而使用标准的系统字体。

- **无视设计图字体：** **不要**根据 Figma 设计图设置特定的 `font-family` 样式。
- **使用系统默认：** 允许应用程序自动使用浏览器或原生平台提供的默认系统字体（例如默认的无衬线体 sans-serif）。
- **保留其他样式：** 您仍应保留其他的排版特征，例如字体粗细（如需要加粗则用 `bold`、`font-bold` 等）、字体大小（字号）和行高。

## 4. 交互状态与代码策略
Figma 设计通常只呈现交互组件的静态快照。您必须推断并实现必要的行为代码。

- **实现交互：** 如果 UI 自然需要交互（例如：按钮、切换开关、标签页、滚动列表、模态框），您必须实现相应的状态管理（例如：React 中的 `useState`）。
- **代码质量与策略：**
  - 编写整洁、富有策略性的代码，避免冗余。
  - **不要**仅仅为了实现某种视觉状态而编写杂乱无章的“意大利面条式”逻辑。
  - 在符合逻辑和减少冗余的情况下，考虑将重复的 UI 元素提取为 `map` 循环或独立子组件。
  - 确保页面交互和用户体验符合标准的 Web/App 开发范式。
